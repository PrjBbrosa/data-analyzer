# Spec Review: Batch-Blocks Redesign
## Verdict
needs revision before plan

## Summary
The redesign is a coherent direction, and the component split broadly matches the requested input-analysis-output workflow. The spec still has several planning blockers around lazy disk files, missing target signals, cancellation, and preset versioning. Those gaps affect both the UI contract and the runner API, so they should be resolved before turning this into an implementation plan.

## Findings
1. **F-1 [must-fix] §3.2/§3.5/§4.3/§7** — Lazy disk files make preview and failure UX underspecified
   - Observation: §3.2 says "+ disk" only adds paths and does not load files, while file list changes still recompute the signal intersection (spec lines 109-114). §3.5 requires a pre-run dry-run task list (spec lines 159-204). §4.3 says `file_paths` are resolved by lazy `loader.load(path)` at run time (spec lines 286-288). §7 says a bad mf4 appears red in the file list with a tooltip and is excluded from task expansion (spec line 403).
   - Issue: These rules cannot all hold unless the spec defines an intermediate state for path-only files. The UI cannot compute signal intersection or an exact dry-run task list for an unloaded 5 GB file, and a run-time read failure has no specified row/task status or continuation rule. Existing `BatchRunner` currently only receives `files` in `__init__` and has no loader dependency (mf4_analyzer/batch.py:91-92); current free-config expansion iterates already-loaded `self.files` only (mf4_analyzer/batch.py:144-150).
   - Suggestion: Specify file states such as `loaded`, `path_pending`, `loading`, and `load_failed`; define whether preview counts are exact, partial, or "unknown until run" for path-only files; add the loader dependency/caching contract; and state whether a disk read failure marks only that file's tasks failed or fast-fails the whole batch.

2. **F-2 [must-fix] §4.2/§4.3/§7** — All target signals missing has no runnable-task rule
   - Observation: §4.2 only describes the case where "some" imported `target_signals` are unavailable and says red chips plus warning should not force-stop running (spec lines 256-258). §4.3 yields missing-signal tasks so `_run_one` can fail them (spec lines 273-280). §7 says a missing-signal task is marked failed with `"missing signal: X"` (spec line 404), but it does not define the case where none of the preset's target signals exist in any current file.
   - Issue: If `target_signals` is non-empty but every requested signal is missing, the run button could appear enabled even though there are zero useful tasks. Existing runner behavior for an empty expansion is a blocked result with `"no matching batch tasks"` (mf4_analyzer/batch.py:99-104), and current-single missing signals are silently skipped during expansion (mf4_analyzer/batch.py:135-142).
   - Suggestion: Decide the UX explicitly: either disable run with an INPUT warning when runnable tasks are zero, or show precomputed failed rows for every missing file/signal pair and allow a no-op run. The task list should have a clear per-task status for this exact scenario.

3. **F-3 [must-fix] §3.5/§4.4/§6.2/§7** — Cancellation path is not implementable yet
   - Observation: §3.5 says the button area switches to a single "interrupt" button during run (spec lines 201-204). §6.2 only describes `task_started`, `task_done`, `task_failed`, and unlock after all done (spec lines 355-369). §4.4 defines `BatchProgressEvent.kind` as only `task_started`, `task_done`, or `task_failed` (spec lines 295-304). §7 only covers closing the dialog during run and says to cancel the worker thread (spec line 405).
   - Issue: The spec does not define the interrupt button handler, a cancel token, a canceled result/status, event emission for the current row, or state restoration after partial completion. Existing `BatchRunner.run` is synchronous and has no cancel argument or check points (mf4_analyzer/batch.py:94-130).
   - Suggestion: Add a cancel contract to `RunnerThread` and `BatchRunner.run`, preferably a `cancel_token` checked between tasks and passed into order analyzers where applicable; add a `task_cancelled` or `run_cancelled` event/result status; define unlock behavior and retained row states after cancellation.

4. **F-4 [should-fix] §4.1** — `AnalysisPreset` is becoming an overloaded run request
   - Observation: §4.1 adds `file_ids`, `file_paths`, and `target_signals` to `AnalysisPreset`, but says they are for `free_config` only (spec lines 208-223). The current dataclass already contains source-specific fields such as `signal`, `rpm_signal`, `signal_pattern`, and `rpm_channel` (mf4_analyzer/batch.py:31-42), and the existing `free_config` factory only writes pattern/RPM/params/output data (mf4_analyzer/batch.py:57-66). §4.2 then excludes `file_ids`, `file_paths`, and output directory from portable JSON (spec lines 234-255).
   - Issue: One object would represent both a portable analysis recipe and a transient UI/run selection. That is sustainable only if invariants are enforced; otherwise, stale `file_ids`, accidental JSON leakage, and invalid source/field combinations become easy footguns.
   - Suggestion: Split the model into a portable `AnalysisPreset` plus a `BatchRunRequest` or `BatchSelection`, or keep one dataclass but add explicit factories/validation and a serialization whitelist that rejects source-incompatible fields.

5. **F-5 [should-fix] §4.2** — Preset JSON should include `schema_version` from v1
   - Observation: The non-goals say no preset migration/version mechanism and "v1 schema can break directly" (spec lines 39-44). §4.2 explicitly defines v1 JSON without `schema_version` (spec lines 225-255). §7 still references schema mismatch handling (spec line 406), and §8 plans tests for missing-field downgrade and bad JSON (spec lines 411-418).
   - Issue: Once users save presets, the loader cannot distinguish old shape, current shape, and future breaking shapes. That makes the first breaking change harder than adding a version now.
   - Suggestion: Add `"schema_version": 1` to newly saved presets, reject unknown versions with the existing unsupported-format toast, and optionally treat missing `schema_version` as v1 only if there is a deliberate backward-compatibility reason.

6. **F-6 [should-fix] §4.4/§10** — Runner compatibility is workable but the signature must be canonical
   - Observation: §4.4 first shows `run(self, preset, output_dir, on_event=...)` (spec lines 292-307), then says the compatible signature accepts both `progress_callback=None` and `on_event=None` and calls the old callback after each `task_done` (spec line 309). §10 repeats that `progress_callback` is retained while `main_window.open_batch` moves to `on_event` (spec line 448). Current `BatchRunner.run` is `run(self, preset, output_dir, progress_callback=None)` and calls that callback after each item (mf4_analyzer/batch.py:94, mf4_analyzer/batch.py:121-122). `MainWindow.open_batch` calls `BatchRunner(self.files).run(preset, output_dir)` with no callback (mf4_analyzer/ui/main_window.py:977-980); observed batch-runner tests also call it with two arguments (tests/test_batch_runner.py:31, tests/test_batch_runner.py:50, tests/test_batch_runner.py:67, tests/test_batch_runner.py:116, tests/test_batch_runner.py:135, tests/test_batch_runner.py:153).
   - Issue: The compatibility claim is workable, and no observed repo caller passes `progress_callback` today, but implementers could accidentally break a positional third-argument caller if they follow the earlier on-event-only prototype.
   - Suggestion: Specify the canonical signature as `run(self, preset, output_dir, progress_callback=None, *, on_event=None, cancel_token=None)` or similar. Keep `on_event` keyword-only, define ordering when both callbacks are passed, and test the legacy positional callback path.

7. **F-7 [should-fix] §8** — Test plan misses the highest-risk behavioral seams
   - Observation: §8 covers preset IO, target-signal expansion, `file_paths` lazy loading, events, missing signals, progress compatibility, and UI widgets (spec lines 409-430).
   - Issue: The plan does not test several behaviors that are underspecified elsewhere: run-time disk load failure, all target signals unavailable, cancellation through button and close-dialog paths, state unlock after cancellation/failure, output directory write fast-fail, JSON version handling, and simultaneous `progress_callback` plus `on_event`.
   - Suggestion: Add focused runner and UI tests for those cases before implementation begins. These tests should pin the decisions requested in F-1 through F-6.

## Section-by-section notes
### §R1 Internal consistency — do sections contradict each other? Does the data model in §4 match the UI behavior in §3? Does §5 component decomposition cover everything §3 describes?
The main inconsistency is path-only disk files: §3.2 says they are not loaded, but the UI still recomputes intersections and §3.5 previews exact tasks. §4.3 pushes file resolution to run time, while §7 expects file-list-level error display. See F-1. The `target_signals` model mostly matches the UI, but imported presets with unavailable chips need the zero-runnable-task rule in F-2. §5 covers the visible UI components from §3, including strip, input, analysis, output, signal picker, task list, and runner thread.

### §R2 Scope sanity — single coherent spec or needs decomposition? Evaluate the 6 PR slices in §9.
This can remain one coherent spec, but it should not become an implementation plan until the must-fix behavior is settled. The six PR slices in §9 are generally sensible: backend first, shell, detail widgets, progress wiring, preset/current-fill wiring, then cleanup. PR 1 is the riskiest because it combines data model, task expansion, event API, and JSON IO; it may need an internal split or very tight tests. PR 4 should explicitly include cancellation once F-3 is resolved.

### §R3 Missed requirements / edge cases — what's missing from §7 (errors)? Specifically: (a) user adds a 5 GB mf4 via "+ 磁盘" — lazy-load deferred; what is the UX if disk read fails at run-time mid-batch? (b) preset's target_signals contain none of current files' channels — how should the per-task status show that? (c) cancellation mid-run — does state restore cleanly? Is there a spec'd cancel path?
§7 does not yet answer these. A run-time disk read failure needs row-level file/task status and a continue-or-fast-fail rule (F-1). If none of the preset signals exist, the spec must decide between disabling run or showing all rows as pre-failed/missing (F-2). Cancellation is only hinted at; there is no cancel token, event kind, result status, or state restoration contract (F-3).

### §R4 Backwards compat claims — §4.4 says run() keeps old progress_callback alongside on_event. Is that workable? Does any caller in the repo actually pass progress_callback? (check main_window.open_batch)
It is workable if the old third positional parameter remains `progress_callback` and `on_event` is added after it, preferably keyword-only. I did not observe a repo caller passing `progress_callback` to `BatchRunner.run`: `open_batch` passes only `preset` and `output_dir` (mf4_analyzer/ui/main_window.py:977-980), and the observed batch runner tests do the same (tests/test_batch_runner.py:31, tests/test_batch_runner.py:50, tests/test_batch_runner.py:67, tests/test_batch_runner.py:116, tests/test_batch_runner.py:135, tests/test_batch_runner.py:153). See F-6.

### §R5 API surface choices — BatchProgressEvent dataclass with kind: Literal[...]: appropriate vs subclasses / pyqt signals? The spec says UI wraps it in pyqtSignal — is the dataclass the right intermediate layer?
The dataclass is the right intermediate layer because `mf4_analyzer/batch.py` is currently GUI-free and the spec preserves that separation. Subclasses would add little value for three event shapes with mostly shared fields. The missing piece is not the dataclass choice; it is the event vocabulary for skipped/unavailable/cancelled states and final run completion. See F-3 and F-7.

### §R6 Data model — §4.1 extends AnalysisPreset in place. Is one dataclass with optional fields (file_ids/file_paths/target_signals only meaningful for free_config) sustainable, or a footgun?
It is a footgun unless the spec adds invariants. The current dataclass is already source-sensitive, and adding file/runtime selection fields makes it carry both portable recipe data and per-run UI state. Splitting recipe from run request would be cleaner; otherwise, factories and serialization should enforce valid combinations. See F-4.

### §R7 YAGNI — anything over-designed? Specifically evaluate lazy-load of disk-added files vs pre-load on add.
Lazy loading disk-added files is not YAGNI for very large mf4 inputs; pre-loading on add would make the dialog feel broken for the 5 GB case named in the review prompt. The over-design risk is pretending lazy files behave like loaded files in preview and signal selection. The right default is lazy load plus explicit pending/unknown UI states, or a cheap metadata probe if the file format supports it. See F-1.

### §R8 JSON schema (§4.2) — "no schema_version" decision: is that fine, or will users with saved presets be locked out on the next breaking change? What is the right default here?
The no-version decision is not fine for a newly introduced saved-preset feature. There is almost no cost to writing `schema_version: 1` now, and it prevents ambiguity on the next breaking change. The right default is to include version 1 in saved files, reject unknown future versions, and only accept missing version as v1 if backward compatibility is intentionally needed. See F-5.

### §R9 Test plan (§8) — adequate coverage? What cases are missing?
The test plan is a good start but not adequate for plan approval. It covers the happy-path decomposition and several UI widgets, but it omits the edge cases most likely to break the redesigned workflow: lazy disk read failure, zero runnable target signals, cancellation, unlock/restore after cancel or failure, output fast-fail, JSON version handling, and dual callback behavior. See F-7.
