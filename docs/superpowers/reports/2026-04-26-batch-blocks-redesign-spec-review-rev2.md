## Verdict
needs revision before plan

## Summary
Rev 2 resolves several Rev 1 blockers by adding file states and load-failure handling in spec Sections 3.2 and 7, schema-version handling in Section 4.2, and cancellation API details in Sections 4.4 and 4.5. It is not ready for planning because Sections 4.1 and 4.2 disagree on whether `signal_pattern` is serialized, Sections 4.3 and 8 disagree on the all-missing target-signal runner result, and Sections 4.4, 4.5, and 6.2 disagree on the `run_finished`/unlock flow. The Section 3.2 file-state recompute rules still align with Section 6.1 at the high level because Section 3.2 says probe completion triggers signal-intersection recompute and disables Run while pending/probing, while Section 6.1 recomputes the task list on file/signal changes.

## Prior Findings Status
- **F-1: partially resolved.** Spec Section 3.2 now defines `path_pending`, `probing`, `probe_failed`, `loading`, and `load_failed`; it says dry-run task lists count only `loaded` files and disable Run while `path_pending`/`probing`. Spec Section 7 says a runtime disk-file full-load failure marks all tasks for that file failed and lets other files continue. The remaining gap is Section 4.3: `_resolve_files` says `file_paths` use `loader.load(path)`, but the spec does not define where `loader` comes from, how `BatchRunner` receives it, or how the cache is tested; that was part of Rev 1 F-1's loader/caching concern.
- **F-2: partially resolved.** Spec Sections 4.2 and 7 now state that when imported `target_signals` are all unavailable, INPUT is marked warning and Run is disabled. However, Section 4.3 still shows `_expand_tasks` yielding a task for each missing target signal so `_run_one` can fail it, while Section 8 expects `target_signals` all absent to produce `BatchRunResult.status == 'blocked'`. The UI rule is present, but the runner/task-expansion rule is internally inconsistent.
- **F-3: resolved.** Spec Section 4.4 adds `task_cancelled`, `run_finished`, `cancel_token`, `BatchRunResult.status == 'cancelled'`, and callback ordering; Section 4.5 specifies the interrupt/close flow and task-boundary cancellation. Section 7 routes both the interrupt button and close-dialog case to Section 4.5. NF-1 below still requires data-flow cleanup, but the Rev 1 F-3 implementability gap is substantively addressed.
- **F-4: partially resolved.** Spec Section 4.1 keeps one `AnalysisPreset` dataclass but adds field ownership, source applicability, serialization whitelist intent, factory validation, and an explicit tradeoff against splitting `BatchRunRequest`. That addresses the Rev 1 F-4 invariant concern in principle. It remains unstable because Section 4.1 says `signal_pattern` is serialized, while Section 4.2's schema omits it; see NF-2.
- **F-5: resolved.** Spec Section 2 now says presets are v1-only and future v2 files are rejected, and Section 4.2 writes `schema_version: 1`, treats missing `schema_version` as v1, and rejects unknown versions. Section 8 adds tests for writing v1, reading missing schema version, rejecting v2, and bad JSON.
- **F-6: resolved.** Spec Section 4.4 defines the canonical signature as `run(self, preset, output_dir, progress_callback=None, *, on_event=None, cancel_token=None)`, preserving the old third positional callback and making `on_event` keyword-only. It also defines ordering when both callbacks are passed, and Section 10 repeats that `progress_callback` compatibility is retained.
- **F-7: resolved.** Spec Section 8 now covers the high-risk cases Rev 1 F-7 called out: lazy `file_paths` load success/failure, all and partial missing target signals, output-directory creation failure, all five progress-event kinds, cancellation, legacy `progress_callback`, dual callback ordering, UI file-state transitions, and UI cancellation.

## New Findings
- **NF-1 [must-fix] Terminal event and unlock flow are inconsistent.** Spec Section 4.4 adds `run_finished` and comments that the UI uses it to release the editing lock, but the `BatchProgressEvent` dataclass still requires per-task fields such as `task_index`, `file_name`, `signal`, and `method` for every kind, with no terminal-event payload rule. Section 4.5 then says the runner emits `run_finished(final_status='cancelled')`, but unlock happens on `BatchRunnerThread.finished`; Section 6.2 omits both `run_finished` and `task_cancelled` and unlocks after "all done." Define the required `run_finished` payload, choose the unlock trigger, and update Section 6.2 to include terminal, cancelled, and blocked flows.
- **NF-2 [must-fix] The field-property table does not match preset serialization.** Spec Section 4.1 marks `target_signals` / `rpm_channel` / `signal_pattern` as free-config recipe fields serialized to preset JSON. Spec Section 4.2's v1 schema includes `target_signals` and `rpm_channel` but omits `signal_pattern`, while its whitelist rule says to extract fields marked serializable in Section 4.1. Decide whether `signal_pattern` is portable preset data, then align Sections 4.1, 4.2, and the Section 8 preset IO tests.
- **NF-3 [must-fix] All-missing target-signal backend behavior has three different contracts.** Spec Section 4.2 says all unavailable imported `target_signals` make INPUT warning and disable Run; Section 7 repeats that behavior. Section 4.3's `_expand_tasks` pseudo-code would still yield missing-signal tasks, while Section 8 says the runner should return `BatchRunResult.status == 'blocked'` when all `target_signals` are absent. Choose one backend contract for direct `BatchRunner` calls and make Sections 4.3, 7, and 8 match it.
- **NF-4 [nit] Section 9's runner slice uses a misleading signature shorthand.** Spec Section 4.4 and Section 10 preserve `progress_callback` compatibility, but Section 9 slice 2 describes the new signature as `run(*, on_event=, cancel_token=)`, which omits the preserved third positional callback. The 8-slice dependency order in Section 9 otherwise has no orphaned boundaries: model, runner, preset IO, UI shell, UI detail widgets, task/progress wiring, toolbar import/export, and cleanup are ordered by their stated dependencies.

## Recommendation
- Define the `file_paths` loader contract in Sections 4.3/4.4: where `loader` is injected, how disk-file cache ownership works, and how tests mock load failure.
- Align the all-missing target-signal behavior across Sections 4.2, 4.3, 7, and 8.
- Align `signal_pattern` serialization across Sections 4.1, 4.2, and 8.
- Define the `run_finished` payload and single unlock trigger, then update Section 6.2 for `run_finished`, `task_cancelled`, `cancelled`, and `blocked` flows.
