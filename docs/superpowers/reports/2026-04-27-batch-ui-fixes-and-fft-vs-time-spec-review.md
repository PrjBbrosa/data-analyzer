## New Issues Found
- severity: blocking; affected plan section: Phase 3 Task 4 Steps 4.1 and 4.3 (plan lines 599-608, 620-628, 668-724). Technical description: the tests require high-precision factors (`1/6` within `1e-9`, rad/s within `1e-6`, and exact `rpm_params()` equality), but the implementation sets `_rpm_factor_spin.setDecimals(4)`. In PyQt5, that rounds stored values: `1/6` becomes `0.1667`, and `60/(2*pi)` becomes `9.5493`, so the proposed tests and exported params will fail. Suggested resolution: use enough decimals for the required tolerances, for example 8-10 decimals, adjust the spinbox width if needed, and keep tests tolerant of UI rounding if exact factors are not a product requirement.
- severity: blocking; affected plan section: Phase 1 Task 2 Steps 2.1 and 2.3 plus Phase 7 Task 10.1 (plan lines 158-170, 254-268, 1531-1535). Technical description: Step 2.1 asserts `_display_frame.sizeHint().height()` grows with chip count, but the implementation only places chips in a `QScrollArea` capped by `setMaximumHeight(96)` and does not set a dynamic height. A runtime model of the snippet kept the frame sizeHint height constant for one and four chips. Suggested resolution: explicitly size the chip scroll area from `min(row_count, MAX_VISIBLE_ROWS) * row_height + margins`, or override the display frame size hint; then assert strict growth until the cap and equality after the cap.
- severity: blocking; affected plan section: Phase 3 Task 5 Steps 5.3-5.4 and current `BatchSheet.apply_preset` source (plan lines 799-825; `mf4_analyzer/ui/drawers/batch/sheet.py` lines 326-335). Technical description: the plan removes `rpm_factor` from `DynamicParamForm` and injects it from `InputPanel.rpm_params()` during `get_preset`, but it does not add the reverse path when applying/importing a preset. Existing `apply_preset` sends `preset.params` only to `_analysis_panel.apply_params()` and applies only the RPM channel to `InputPanel`, so imported presets with `params["rpm_factor"]` will silently reset to the input panel default on the next save/run. Suggested resolution: add `InputPanel.apply_rpm_factor()` or `apply_rpm_params()`, call it from `BatchSheet.apply_preset`, and add a round-trip test.
- severity: warning; affected plan section: Phase 5 Task 8 Steps 8.2-8.3 (plan lines 1233-1247). Technical description: Step 8.2 says the ceiling-overflow test should fail, while Step 8.3 says the existing per-task handler is already verified and the test should pass after Step 7.3 alone. Source confirms the handler exists (`mf4_analyzer/batch.py` lines 227-260), so the red/green sequence is internally inconsistent. Suggested resolution: change Step 8.2 expected result to PASS after Step 7.3, or move the test earlier if a red test is required.
- severity: minor; affected plan section: Phase 6 Task 9 Step 9.3 and current `BatchSheet` source (plan lines 1388-1397; `mf4_analyzer/ui/drawers/batch/sheet.py` lines 36-40 and 198-207). Technical description: the plan adds the `fft_time` button label as `FFT vs Time`, but does not update `_METHOD_LABELS`, so the pipeline summary will fall back to the raw key `fft_time`. Suggested resolution: add `"fft_time": "FFT vs Time"` to `_METHOD_LABELS`.
