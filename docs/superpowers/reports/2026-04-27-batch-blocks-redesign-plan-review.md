# Plan Review: 2026-04-27-batch-blocks-redesign

**Date:** 2026-04-26
**Reviewer:** Codex plan-review pass
**Verdict:** needs revision before execution

## Executive Summary
The wave structure broadly matches the approved spec, and the backend/UI split is mostly in a workable order. However, several execution details conflict with the spec: Wave 6 can full-load disk MF4 files on the UI thread during dry-run, Wave 2's runner pseudocode violates the progress callback contract, and disk load failures can collapse into a blocked run instead of per-task failures. The plan also misses several spec §8 test cases and leaves some cross-wave UI APIs implicit. These issues should be fixed in the plan before squad execution begins.

## Findings

1. [must-fix] Wave 6 dry-run can full-load disk files before the worker thread — Spec §3.2 says "+ 磁盘…" must not do full loading and should only probe `MDF(path).channels_db.keys()` until lazy runtime loading; spec §6.1 says the task list is recomputed as dry-run preview from configuration state. Plan W6 instead says "On 运行: build dry-run task list (re-run `_expand_tasks` synchronously)" while Plan W2 defines `_resolve_files` so `file_paths` call `self._loader(path)`. Side by side, the spec keeps sample loading out of the dialog/dry-run path, but the plan's W6+W2 combination loads samples synchronously before `BatchRunnerThread.start()`, which reintroduces the large-file freeze the redesign is meant to avoid.

2. [must-fix] Disk load failures can be reported as zero tasks instead of failed task rows — Spec §3.2 and §7 say a runtime full-load failure for a disk path marks all tasks for that file as `task_failed` with an error such as "cannot load mf4: ...", while other files continue and the run does not fast-fail. Plan W2's `_resolve_files` yields `_LoadFailure`, but W2's `_expand_tasks` phase 1 ignores `_LoadFailure` when computing `has_any_runnable`; if every selected disk file fails to load, `_expand_tasks` returns no tasks and `run()` reports `BatchRunResult(status='blocked', blocked=['no matching batch tasks'])`. That is a confirmed plan/spec mismatch, not just a hypothesis, because the W2 pseudocode has no path that emits `task_failed` rows for an all-failed disk selection.

3. [must-fix] `progress_callback` semantics drift from the approved event contract — Spec §4.4 says when both callbacks are supplied, each `task_done` event calls `on_event(event)` first and then `progress_callback(index, total)`; spec §8 separately says the legacy progress callback call count must equal completed task count. Plan W2's brief repeats that promise, but the W2 `run()` pseudocode places `if progress_callback: progress_callback(index, total)` after the `try/except`, so it runs for failed tasks as well as successful tasks. The fix is to call `progress_callback` only in the successful `task_done` branch after `on_event(task_done)`, and add a failing-task regression test.

4. [must-fix] Wave 4 deletion does not update all current references to `batch_sheet.py` — Spec §10 requires deleting `mf4_analyzer/ui/drawers/batch_sheet.py` and replacing it with `drawers/batch/`, and Plan W4 updates `main_window.py` plus `tests/ui/test_drawers.py`. A repository reference check shows `tests/ui/test_order_smoke.py` still patches `mf4_analyzer.ui.drawers.batch_sheet.BatchSheet`; after W4 deletes the module, that test will fail unless W4 also updates it or provides a temporary compatibility shim. This is exactly the dependency-order issue the review question calls out: something still references the deleted module after the deletion wave.

5. [should-fix] UI unlock is wired to the result signal instead of the guaranteed thread-finished signal — Spec §4.4 and §6.2 explicitly say `run_finished` is semantic only and editing must unlock via `BatchRunnerThread.finished` because Qt emits it even if the runner exits unexpectedly. Plan W6 says `On finished_with_result: unlock_editing(), swap buttons back, toast`, and its `BatchRunnerThread.run()` has no `try/finally` or exception-to-result wrapper. If `runner.run()` raises before emitting `finished_with_result`, the dialog can remain locked, which the spec specifically designed around.

6. [should-fix] Task list/progress UI is under-specified relative to spec §3.5 — Spec §3.5 requires the collapsed header to show "N 任务待执行 · M 输出" and the running state to show "进度 i / N", a progress bar, and an ETA-style remaining-time text. Plan W6's `TaskListWidget` only specifies rows, icons, `apply_dry_run(tasks)`, and `on_event(event)`, and its tests only assert row count and icon transitions. The self-review mapping "§3.5 task list (run-time states) → Wave 6" is therefore incomplete for output count, progress bar, and remaining-time behavior.

7. [should-fix] Spec §8 test coverage is materially incomplete despite the self-review claim — Spec §8 lists runner tests for multi-signal `target_signals`, callback counts, cancellation completeness, and UI tests for PipelineStrip status transitions, SignalPicker focus/ESC collapse, file `path_pending → loaded`, run disabled while probing, TaskList collapse, cancel UI, and `apply_preset` missing-signal styling. Plan W2 tests do not include a normal multi-signal expansion test or a failed-task `progress_callback` count test; W5 tests do not verify focus/ESC collapse, `path_pending → loaded`, or run-disabled-while-probing; W6 tests do not cover collapse or the sheet-level cancel path; W7 tests cover `free_config` apply but not `current_single` apply or partial/all unavailable target-signal behavior. This contradicts Plan self-review's "§8 test plan → tests created in Waves 1-6" mapping, and W7 adds tests as well.

8. [should-fix] Cross-wave UI accessor and `apply_preset` contracts are implicit — Spec §6.4 says "从当前单次填入" fills the file list, target signal, method, params, RPM channel, and time range. Plan W7 tests call `sheet.method()`, `sheet.selected_signals()`, and `sheet.rpm_channel()`, and W7's `apply_preset` uses missing-signal red-marking, but Plan W5 never defines these `BatchSheet` or panel accessors, nor does W7 mention restoring the time-range field for `current_single`. The plan should name the panel/sheet methods and slots that W7 consumes and should include current-single plus time-range round-trip tests.

9. [nit] Default loader contract differs from the approved spec and should be clarified — Spec §4.3 defines `_default_loader(path)` as importing `load_file` from `mf4_analyzer.io.loader` and returning `load_file(path)`. Plan W2 instead says the default loader calls `DataLoader.load_mf4` and wraps the result in `FileData`, which appears aligned with the current repo API but not with the approved spec text. This is not necessarily wrong, but the plan should either match spec §4.3 or explicitly record the intentional adaptation so specialists do not implement different loader contracts.

## Section-by-Section Notes

### Q1: Spec Coverage
The plan's self-review is mostly accurate for the large sections: spec §4.1 maps to W1, §4.2 to W3 plus W7, §4.3 and §4.4 to W2, §4.5 to W2/W6, §5 to W4-W6, §6.3 to W7, §9 to the wave plan itself, and §10 to W4/W8. The uncovered or only partially covered areas are spec §3.5's output-count/progress-bar/ETA behavior, spec §3.2's strict metadata-only dry-run behavior for disk files, spec §6.2's requirement that unlock use `QThread.finished`, spec §6.4's time-range restore for `from_current_single`, and several spec §7 imported-preset/missing-signal UI states. The checklist entry "§8 test plan → tests created in Waves 1-6" is inaccurate because W7 also creates tests and several §8 cases remain unplanned.

### Q2: Wave Dependency Order
The general wave order is coherent: W1 adds the preset fields needed by W2/W3/W5, W2 produces `BatchProgressEvent` for W6, W3 produces JSON IO for W7, W4 creates the package that W5-W7 extend, and W8 does final cleanup. The false dependency is W4's deletion of `mf4_analyzer/ui/drawers/batch_sheet.py`: Plan W4 updates `main_window.py` and `tests/ui/test_drawers.py`, but current `tests/ui/test_order_smoke.py` still references the old module path after W4. Another order risk is W6's reuse of `_expand_tasks` for dry-run; because W2 makes `_expand_tasks` resolve `file_paths` through the loader, W6 consumes a backend API in a way that violates the earlier metadata-only UI contract.

### Q3: Type / Signature Consistency
The core runner signatures are mostly consistent across waves: W2 defines `BatchRunner.__init__(files, loader=None)` and `run(preset, output_dir, progress_callback=None, *, on_event=None, cancel_token=None)`, and W6's `BatchRunnerThread.run()` calls `runner.run(self._preset, self._output_dir, on_event=self.progress.emit, cancel_token=self._cancel_token)`, which matches the keyword-only contract. W7's `load_preset_from_json(path)` and `save_preset_to_json(preset, path)` calls match W3. The inconsistencies are behavioral rather than syntactic: W2's `progress_callback` placement violates the declared callback semantics, W2's default-loader description differs from spec §4.3, and W7 assumes sheet accessors such as `method()`, `selected_signals()`, and `rpm_channel()` that W5 never defines.

### Q4: Test Coverage vs Spec §8
Covered well: preset JSON round-trip, whitelist, schema version handling, corrupt JSON, all-missing target signals, partial-missing target signal failure, disk load success/failure with another good file, output directory create failure, basic event emission, legacy callback success path, dual-callback ordering on success, method parameter rendering, basic signal search/selection, probe failure, and basic task icon updates. Missing or weak: normal multi-signal expansion, all-disk-load-failed task-failure behavior, failed-task progress-callback count, cancellation output completeness/no half-written file check, PipelineStrip 0/1 status transitions, SignalPicker outside-click/ESC collapse, file `path_pending → probing → loaded`, run disabled while probing, TaskList collapse/header/progress bar/output count, sheet-level cancel UI, `apply_preset(current_single)`, partial/all unavailable target-signal red/warning/disabled states, and the 5GB metadata-probe performance smoke from spec §8.

### Q5: Code Completeness / No Placeholders
Most waves give enough class names, methods, signals, and tests for specialists to proceed, and the W4 placeholder shell is acceptable because W5-W7 explicitly replace it. The incomplete areas are the cross-wave UI API contract for `BatchSheet` and panels, the lack of concrete TaskList header/progress API, and the lack of explicit behavior for missing-signal chips and time-range application. Minor note: Plan W4's `PipelineCard` code creates `badge_label` but does not add it to a visible layout, even though spec §3.1 requires card status badges; this should be fixed when revising W4.

### Q6: End-to-End Risks
Confirmed risk: W6's dry-run path can call W2's disk loader and full-load samples on the UI thread, undermining the spec §3.2 metadata-only probe requirement. Confirmed risk: all selected disk files failing to load can produce a generic blocked result instead of per-task failures, because W2's runnable check ignores `_LoadFailure`. Confirmed risk: unexpected runner exceptions can leave the dialog locked because W6 uses `finished_with_result` rather than `QThread.finished`. Hypothesis: metadata-only probing with `MDF(path).channels_db.keys()` is likely viable for channel enumeration without sample decoding, but the plan should still include the spec §8 5GB smoke/performance check and ensure the worker closes the MDF object. Hypothesis: re-entrant runs are probably prevented by W6's `_running` flag and button swap, but the plan should state that `_running` is set before thread start and that the Run action is disabled synchronously to avoid double-click races.

## Verdict Rationale
The plan should not execute as-is because several must-fix issues would produce behavior the approved spec explicitly rejected: UI-thread full loads, incorrect legacy progress semantics, and missing per-task failures for disk-load errors. Once those are corrected and the §8 coverage gaps are added to the wave briefs, the overall wave decomposition should be viable.
